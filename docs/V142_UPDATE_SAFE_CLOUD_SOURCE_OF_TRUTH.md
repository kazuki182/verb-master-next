# Verb Master V142 ZIP更新対策・クラウド本命保存設計

## 目的
ZIPをGitHubに上書きしてVercelへ再デプロイしても、ユーザーの画像・ニックネーム・目標日・学習記録・課金状態が0/124へ戻らないようにする。

## 根本原因
過去版では、クラウド復元に失敗した場合でもローカルログインへ逃げられたため、端末側に新規の空データが作られることがあった。ユーザーから見ると「ZIP更新のたびに最初から」に見える。

## 新しい原則
- `user_progress_backups` を本命保存先にする。
- `localStorage` は表示速度を上げるためのキャッシュだけにする。
- Supabaseが設定済みの本番運用では、クラウド復元できない状態でホーム画面を表示しない。
- ローカル専用ログイン・ローカル専用登録に逃がさない。
- 空の端末データがクラウドの実データを上書きしないよう、SQL側でも防止する。

## アプリ側の動き
1. ログイン時に必ずクラウドログインを試す。
2. クラウドログイン成功後、`user_progress_backups` から復元する。
3. 復元できるまでホーム画面の0/124を表示しない。
4. Supabase設定済みなのにクラウドログインできない場合、ローカルログインへ逃がさずエラー表示する。
5. 変更があったらクラウドへ保存する。

## SQL側の防御
`vm_upsert_progress_backup` で既存クラウドデータのスコアを計算し、既存スコアがあるのに新データが空なら保存をスキップする。

例：
- クラウド：XPあり、学習記録あり
- 端末：ZIP更新後の空キャッシュ XP0 / 0語

この場合、端末の空データはクラウドを上書きできない。

## 導入後の確認
1. Supabase SQL Editorで `supabase/V142_UPDATE_SAFE_CLOUD_SOURCE_OF_TRUTH.sql` を実行。
2. GitHubへV142 ZIPを上書き。
3. VercelでReady確認。
4. アプリでログアウト。
5. 同じID/パスワードでログイン。
6. 画像・ニックネーム・目標日・学習記録を変更。
7. 画面更新。
8. 再デプロイ後に再ログイン。
9. 0/124に戻らないことを確認。

## 運用ルール
今後のZIP更新では、動詞データやUIだけを変えても、以下は変更しない。
- `cloud_accounts`
- `user_progress_backups`
- `user_progress_backup_events`
- `vm_login_cloud_account`
- `vm_fetch_progress_backup`
- `vm_upsert_progress_backup`

保存基盤を変える時は、必ずSQLと設計書を同時更新する。
